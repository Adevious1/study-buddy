# Whole-app audit — 2026-06-11

A four-dimension sweep (security/authz, server robustness, client UX, product
completeness) of the app as of `main` @ `9d75fe6` (SP1–SP8 merged). This is the
reference list for follow-up work; items marked **✅ fixed** were addressed the
same day in the easy-fixes pass. Severity is about impact on a *real family
using the product*, not code style.

The audit's headline: the engineering core (auth, `childContext` ownership,
voice relay, entitlement) is solid. The big holes are product-lifecycle gaps —
no reverse flows (delete/edit/cancel), no compliance surface, no observability.

---

## P0 — blocking for any real launch

### 1. No deletion of anything (COPPA-shaped) — ✅ fixed (SP9, PR #6)
No delete-child, no delete-account, no data-deletion path. `routes/me.ts` has
POST endpoints only; zero DELETE routes anywhere. For a kids' product storing
voice transcripts and photos of children's work, verifiable parental deletion
is a legal requirement, not a feature. Also means seat count can never
decrease. Suggested scope: one "account lifecycle" subsystem — delete child
(cascade sessions/snapshots/profiles + Stripe seat decrement), delete account,
edit child, PIN reset, all behind `childContext`/guardian authz.

### 2. No legal/compliance surface — ✅ largely fixed (SP9, PR #6)
No privacy policy, terms, or parental-consent page anywhere in `apps/web`.
Needed before any non-family user touches the product.
*SP9 shipped public `/privacy` + `/terms`, the login consent line, and a
recorded parental-consent checkbox (`children.consent_at`); the page copy is
solid placeholder pending counsel review.*

### 3. Zero production observability
JSON logs to stdout only (`apps/server/src/logging.ts`); no error tracking, no
analytics, no alerting. A misbehaving Pip or silently failing sessions are
invisible. Minimum: an error tracker (e.g. Sentry) on server + web, and a
counter on session outcomes (completed/abandoned/fallback-recap).

### 4. Stripe webhook: no event dedup / ordering guard
`routes/stripeWebhook.ts:35-39` applies events in arrival order; a late stale
event can overwrite newer subscription state (wrongly locking out a paying
guardian). Partially documented as accepted in `SP5-manual-smoke.md`, but it
gates the core product. Fix: persist processed `event.id`s, or refuse to move
`current_period_end` backwards.

## P1 — real holes, contained scope

### 5. Boot-time env validation — ✅ fixed (partial)
- ✅ `voice/voiceRoute.ts` defaulted `GEMINI_API_KEY` to `''` → cryptic
  mid-session failure; now fails fast in prod.
- ✅ `lib/stripe.ts` `APP_URL` defaulted to localhost → broken Stripe redirects
  in prod; now fails fast in prod.
- ⬜ Remaining: a single consolidated boot check (DATABASE_URL, Google creds,
  STRIPE_* group) instead of per-module guards.

### 6. Snapshot upload decoded base64 before size check — ✅ fixed
`voice/relay.ts` `handleSnapshot` called `Buffer.from(data)` before the size
check; oversized payloads now rejected by base64-length bound first.

### 7. No graceful shutdown / unhandled-rejection handler
`index.ts` has no SIGTERM/SIGINT handling; restarts orphan in-progress
sessions and Gemini connections. Several relay paths use fire-and-forget
`void` async (`voiceRoute.ts:40,48`) with no process-level rejection logging.

### 8. No rate limiting or body-size limits
No global or per-endpoint limits anywhere (`index.ts`). PIN verify, child
create, and the webhook are spammable; a huge JSON body is buffered before
Zod sees it.

### 9. PIN lockout is in-memory, per-instance
`lib/pinLockout.ts` — resets on restart, not shared across instances. Already
noted in-code as a follow-up; needs Postgres persistence before multi-instance.

### 10. No snapshot retention policy
`session_snapshots` (`bytea`) grows forever; photos of children's work need a
documented retention window + automated cleanup (pairs with item 1).

### 11. Misc robustness (small, batchable)
- `relay.ts` `handleAudio` / `handleControl` sends lack try-catch; a throwing
  `session.sendAudio` propagates uncaught.
- `geminiSession.ts:104` `close()` doesn't await the underlying close.
- Concurrent `reconnect()` possible if Gemini closes again during backoff
  (`relay.ts:112`).
- No index on `subscriptions.stripe_customer_id` (webhook hot path scans).
- Zod issues leaked verbatim in 400s (`me.ts:85`) — minor info disclosure.
- Snapshot save-failure log lacks session/child context (`relay.ts:247`).

## P2 — product experience gaps

### 12. Assignments/plans are seed-only
`routes/assignments.ts` / `subjects.ts` are read-only; no UI or API creates
plans. A fresh real account has nothing real to study, and the voice
subject-picker is a hardcoded 4-item list (`VoiceRoute.tsx`). Needs its own
brainstorm (guardian assignment authoring? curriculum presets?).

### 13. Payment failure → hard lockout, no warning
`past_due` → child locked out of `/app` with no email, no in-app grace banner,
no path but the Stripe Portal. Pairs with item 15.

### 14. No child-profile editing — ✅ fixed (SP9, PR #6)
`routes/children.ts` is GET-only — name/grade/birthdate are write-once at
onboarding.

### 15. No transactional email
No receipts, trial-ending warnings, or payment-failed alerts (no email service
integrated at all).

### 16. Visible leftovers — ✅ fixed
- ✅ Hardcoded "Tuesday · April 22" rendered live (`HomeRoute.tsx:61`,
  `DashboardRoute.tsx:187`) → real date.
- ✅ Dashboard "Pip's noticing" card was static copy (`DashboardRoute.tsx:363`)
  while real `learning_profiles` data sat unread → now wired via
  `getLearningProfile` with a getting-to-know-you fallback.
- ✅ `Button` had no `disabled` prop (call sites faked it with opacity, leaving
  buttons clickable) → real attribute; `submit()` guards meant no data bug,
  but a11y/UX was wrong.
- ✅ Undersized touch targets for K-5 fingers (36px back/close buttons, 26px
  color swatches) → ≥44px hit areas.

### 16b. New findings from the 2026-06-12 smoke runs — ✅ both fixed (PR #7)

- **`/dashboard` has no mobile layout.** It's the desktop tree by design, but a
  guardian opening it on a phone gets the fixed sidebar consuming the screen
  and the main column crushed — observed live during the SP7/SP8 smoke.
  *Fixed: below the `md` breakpoint the dashboard shows a friendly "needs a
  bigger screen" notice with a back-to-app button (a true responsive collapse
  remains future work).*
- **Empty-transcript recap confabulation.** A ~13-second session with 0
  transcript turns still produced a generated recap claiming invented
  achievements ("solved 3 of 4", 3 stars).
  *Fixed: `generateRecap` uses the modest fallback — without calling Gemini —
  below 4 transcript turns or when the child never spoke.*

### 17. Dashboard ↔ data completeness
Learning-profile *traits* are still not shown on the dashboard (only the note
is, post-fix; full traits render lives on the child's `/app/profile`). No
seeded `session_snapshots`, so the dashboard snapshot panel is empty until a
real SP7 session.

## P3 — deployment honesty

### 18. docker-compose is dev-only
`docker-compose.yml` uses `target: dev` + bind mounts throughout; the prod
Dockerfile stages (server `bun src/index.ts`, web nginx) are never exercised,
and prod has no migrate/seed story. Blocks the P1 prod-deploy backlog item.

### 19. "Targets iOS/Android" is aspirational
No Capacitor/React Native/PWA artifacts (not even a manifest or service
worker). It's a responsive web app today; either build the wrapper or soften
the claim.
